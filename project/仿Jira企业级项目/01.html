npx create-react-app jira --template typescript

npm start

src/react-app-env.d.ts // 引入预先定义好的typescript类型

src/reportWebVitals.ts // 埋点上报

src/setupTests.ts // 配置单元测试

public/manifest.json // 用来配置pwa

# 配置 eslint、 prettier 和 commitlint 规范工程

配置tsconfig.json

{
  "compilerOptions": {
    "baseUrl": "./src"
    // ...
}

在配置baseUrl之前我们引入文件是使用下面这种../../之类的

import ajax from '../../utils/ajax'

配置之后

import ajax from 'utils/ajax'

这样方便了很多。

# Prettier

在团队编写代码的时候每个人可能有每个人的风格，每个人用的IDE可能不同，即便用的相同的IDE配置可能也不同。所以导致代码风格不尽相同。比如下面这种挨揍的代码格式，就会让人发疯👊：

我们可以使用Prettier来统一代码风格，提高开发效率。废话不多说，来看看怎么使用Prettier。Prettier官方文档 (opens new window)

安装

npm install --save-dev --save-exact prettier

创建一个配置文件

echo {}> .prettierrc.json

创建 .prettierignore 文件，用于指定哪些文件不需要被格式化

# Ignore artifacts:
build
coverage

在终端执行如下命令：

npx prettier --write .

发现上面混乱的代码被格式化了

上面的这种格式看着不那个找揍了😀

但是让每个人都去时不时的执行上面这个命令显然可操作性太差。有没有什么办法可以让它自动执行呢？我们可以使用 Pre-commit Hook (opens new window)

在终端执行如下命令：

npx mrm lint-staged

package.json文件中会新增如下代码








 





 
 


 
 
 


{
  "name": "test",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "prepare": "husky install"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "devDependencies": {
    "husky": "^6.0.0",
    "lint-staged": "^10.5.4",
    "prettier": "2.2.1"
  },
  "lint-staged": {
    "*.{js,css,md}": "prettier --write"
  }
}

这样当我们每次执行commit的时候，代码就会被自动格式化。

假设我们在项目中使用了eslint，我们需要安装eslint-config-prettier，这是因为prettier和eslint可能会有冲突。

npm install --save-dev eslint-config-prettier

然后在package.json文件中修改eslint的配置项，extends数组最后添加prettier






 




{
  "eslintConfig": {
    "extends": [
      "react-app",
      "react-app/jest",
      "prettier"
    ]
  }
}

# commitlint

安装

npm install --save-dev @commitlint/{config-conventional,cli}

在终端执行如下命令

echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js

然后在终端再执行如下命令（在上面Prettier配置基础上需要执行的命令）

# Add hook
npx husky add .husky/commit-msg 'npx --no-install commitlint --edit "$1"'

单独使用commitlint时需要执行的命令：

# Install Husky v6
npm install husky --save-dev
# or
yarn add husky --dev

# Activate hooks
npx husky install
# or
yarn husky install

# Add hook
npx husky add .husky/commit-msg 'npx --no-install commitlint --edit "$1"'

此时当我们再提交代码时就需要遵循一定的格式才行，随意提交会提交失败。

要遵循的格式（冒号后面要加空格）：

git ci -m <type>[optional scope]: <description>

type ：用于表明我们这次提交的改动类型，是新增了功能还是修改了测试代码又或者是更新了文档

optional scope：一个可选的修改范围。用于标识此次提交主要涉及到代码中哪个模块。

description：描述此次提交的主要内容

示例：

git ci -m 'chore: 配置commitlint'

type的取值如下：

  • build：主要目的是修改项目构建系统(例如 glup，webpack，rollup 的配置等)的提交
  • ci：主要目的是修改项目继续集成流程(例如 Travis，Jenkins，GitLab CI，Circle等)的提交
  • docs：文档更新
  • feat：新增功能
  • fix：bug 修复
  • perf：性能优化
  • refactor：重构代码(既没有新增功能，也没有修复 bug)
  • style：不影响程序逻辑的代码修改(修改空白字符，补全缺失的分号等)
  • test：新增测试用例或是更新现有测试
  • revert：回滚某个更早之前的提交
  • chore：不属于以上类型的其他类型(日常事务)

# 对比常见 Mock 方案 配置 JSON SERVER

在真实开发环境中，前端和后端是并行开发的，这样可以极大提高开发效率，节省成本，所以前期是没有后端接口可用的。

这个时候前端需要的数据需要自己Mock，这里对比下常见的Mock方案，以便开发的时候选择更合适的方案。

# 代码入侵

直接在代码中写死Mock数据，或者请求本地的JSON文件

优点：无

缺点：

  1. 与其他方案比mock效果不好
  2. 与真实环境的server环境切换麻烦

# 请求拦截

如Mock.js原理是重写httprequest的属性

优点：

  1. 与前端代码分离
  2. 可生成随机数据

缺点：

  1. 数据都是动态生成的假数据，无法真实模拟增删改查的情况
  2. 只支持ajax，不支持fetch

# 接口管理工具

代表：rap、swagger、moco、yapi

优点：配置功能强大，接口管理与Mock一体，后端改接口Mock也跟着更改，可靠。

缺点：

  1. 配置复杂依赖后端，可能会出现后端不愿意出手，后者等后端配置完了，接口也开发出来的情况
  2. 一般作为大团队的基础建设存在，没有这个条件的话慎重考虑。

# 本地node服务器

代表：json-server (opens new window)

优点：

  1. 配置简单、json-server甚至可以零代码30秒启动一个REST API SERVER
  2. 自定义程度高，一切尽在掌握
  3. 增删改查真实模拟

缺点：与接口管理工具相比，无法随着后端API的修改而自动修改。

什么是REST API呢？一句话总结URI代表资源、对象，METHOD代表行为，如：

GET /list  // 列表
POT /user // 增加
PUT /list/1 // 替换
PATH /list/2 // 修改
DELETE /list/1 // 删除

推荐使用 json-server 这种方式来mock数据。下面我们简单来说下json-server如何使用。

全局安装

npm install -g json-server

或者局部安装

npm install json-server -D

在项目根目录新建一个__json_server_mock__（这样起名的是为了告诉其他开发者这个文件和项目是无关的）文件，并在这个文件中创建一个json文件，如db.json，输入如下内容

{
  "user": []
}

启动

json-server --watch db.json

配置package.json文件，注意：如果是全局安装的json-server要去掉npx

"scripts": {
  // ...
  "server": "npx json-server __json_server_mock__/db.json --watch"
}

# 【注意了】大家不用再手动引入 React 了

以前我们需要在每个React组件中都要手动引入React

import React from 'react'

这样做的原因是因为JSX是语法糖，下面这个组件在编译后

import React from 'react'

function App() {
  return <h1>hello</h1>
}

会被转换成

import React from 'react'

function App() {
  return React.createElement('h1', null, 'hello')
}

每个组件在编译后都会使用React，所以每个组件都需要手动引入React。

这样是不是感觉很麻烦😭，现在我们不需要手动在引入React了😄。

去年年底plugin-transform-react-jsx发布了新版本，从v7.9.0开始，就不同手动引入React了，转换变成了这样，详见Introducing the New JSX Transform (opens new window)

function App() {
  return <h1>Hello World</h1>;
}

编译成：

// Inserted by a compiler (don't import it yourself!)
import {jsx as _jsx} from 'react/jsx-runtime';

function App() {
  return _jsx('h1', { children: 'Hello world' });
}

注意

create-react-app v4.0版本开始支持，之前的版本并不支持这种转换。

在这个工程中并不是使用tsc将ts编译成js的，使用的是babel编译的ts，这样可以用ts类型检测 + babel编译一切，可以降低开发/配置成本。
